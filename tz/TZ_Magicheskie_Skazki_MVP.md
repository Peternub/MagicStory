# ТЗ: B2C SaaS «Магические Сказки»

## 1. Описание продукта

`Магические Сказки` — веб-приложение для родителей, которое генерирует персонализированные сказки для детей на основе данных о ребенке и жизненной ситуации, а затем автоматически создает аудио-озвучку.

Целевая аудитория:

- Родители детей 3–10 лет
- Русскоязычные пользователи РФ

Основная ценность:

- Быстрое создание персональной сказки под конкретную ситуацию
- Озвучка на русском языке
- Удобное хранение истории сказок в личном кабинете

## 2. Цель MVP

Запустить минимально жизнеспособный продукт, который позволяет:

- зарегистрироваться;
- добавить одного или нескольких детей;
- создать сказку по теме;
- получить текст и аудио;
- прослушать и сохранить результат в истории.

В MVP не включать:

- сложные терапевтические обещания;
- генерацию изображений;
- сложную геймификацию;
- многоуровневые тарифы;
- мобильное приложение.

## 3. Технологический стек

Frontend:

- Next.js `App Router`
- React
- Tailwind CSS

Backend:

- Next.js Server Actions / Route Handlers

База данных и инфраструктура:

- Supabase PostgreSQL
- Supabase Auth
- Supabase Storage

AI:

Платежи:

## 4. Основные пользовательские сценарии

### 4.1 Регистрация и вход

Пользователь может:

- зарегистрироваться по email/password;
- войти в систему;
- выйти из системы;
- восстановить доступ через стандартный flow Supabase Auth.

### 4.2 Управление профилем ребенка

Пользователь может:

- создать профиль ребенка;
- редактировать профиль;
- удалить профиль.

Поля профиля ребенка:

- `name`
- `age`
- `interests`
- `fears`
- `additional_context` — дополнительная информация для персонализации

Ограничения:

- у одного пользователя может быть несколько детей;
- имя обязательно;
- возраст обязателен и должен быть в диапазоне `3–10`.

### 4.3 Создание сказки

Пользователь:

- выбирает ребенка;
- вводит тему или проблему дня;
- запускает генерацию.

Система:

- формирует промпт на основе данных ребенка и темы;
- создает текст сказки;
- после успешной генерации текста запускает генерацию аудио;
- сохраняет результат;
- показывает статус обработки;
- после завершения открывает страницу сказки.

Примеры тем:

- не хотел убирать игрушки;
- боялся спать один;
- поссорился с другом;
- не хотел идти в садик.

### 4.4 Просмотр истории

Пользователь может:

- видеть список созданных сказок;
- открывать отдельную сказку;
- читать текст;
- слушать аудио;
- удалять сказку.

## 5. Функциональные требования

### 5.1 Авторизация

- Использовать Supabase Auth.
- Доступ к данным только после авторизации.
- Все пользовательские данные должны быть защищены через `RLS`.

### 5.2 Генерация текста

Система должна:

- генерировать сказку на русском языке;
- учитывать имя, возраст, интересы, страхи и тему;
- поддерживать добрый, безопасный, понятный для ребенка стиль;
- не использовать агрессивные, пугающие, травмирующие формулировки;
- возвращать текст длиной примерно `800–2000` слов, параметризуемо.

Требования к промпту:

- главный герой — ребенок или его сказочный аналог;
- сюжет должен мягко подводить к полезному выводу;
- история должна быть возрастно-адаптированной;
- запрещены медицинские, психотерапевтические и диагностические утверждения.

### 5.3 Генерация аудио

- После получения текста система отправляет текст в TTS.
- Результат сохраняется в `Supabase Storage`.
- В БД сохраняется публичный или подписанный URL.
- Если TTS завершился ошибкой, текст сказки все равно должен сохраниться.

### 5.4 Статусы генерации

Процесс генерации должен быть асинхронным и иметь статусы:

- `pending`
- `text_generating`
- `text_ready`
- `audio_generating`
- `completed`
- `failed`

Если произошла ошибка:

- сохранять сообщение об ошибке в безопасном виде;
- не показывать пользователю технические детали провайдера;
- показывать понятное сообщение: `Не удалось создать сказку. Попробуйте еще раз.`

### 5.5 Лимиты

В MVP ввести простую модель лимитов:

- у пользователя есть `stories_balance`;
- каждая успешная генерация списывает `1` единицу;
- при неуспешной генерации списание не производится.

## 6. Нефункциональные требования

- Интерфейс адаптивный: desktop + mobile.
- Основной язык интерфейса: русский.
- Все секреты хранить только в `.env`.
- Никаких вызовов внешних AI API с клиента.
- Все запросы к провайдерам выполнять только на сервере.
- Обрабатывать ошибки всех внешних API.
- Код должен быть модульным: UI отдельно, бизнес-логика отдельно, интеграции отдельно.
- Использовать `TypeScript`.
- Логи ошибок должны быть на сервере.

## 7. Архитектура данных

Ниже улучшенная схема.

### Таблица `profiles`

- `id uuid pk references auth.users(id)`
- `email text`
- `subscription_status text default 'free'`
- `stories_balance integer default 3`
- `created_at timestamptz default now()`
- `updated_at timestamptz default now()`

### Таблица `children`

- `id uuid pk`
- `user_id uuid references profiles(id) on delete cascade`
- `name text not null`
- `age integer not null`
- `interests text`
- `fears text`
- `additional_context text`
- `created_at timestamptz default now()`
- `updated_at timestamptz default now()`

### Таблица `stories`

- `id uuid pk`
- `user_id uuid references profiles(id) on delete cascade`
- `child_id uuid references children(id) on delete cascade`
- `theme text not null`
- `title text`
- `text_content text`
- `audio_url text`
- `status text not null default 'pending'`
- `provider_llm text`
- `provider_tts text`
- `error_message text`
- `created_at timestamptz default now()`
- `updated_at timestamptz default now()`

### Таблица `usage_events`

- `id uuid pk`
- `user_id uuid references profiles(id) on delete cascade`
- `story_id uuid references stories(id) on delete set null`
- `event_type text not null`
- `amount integer not null`
- `created_at timestamptz default now()`

Назначение:

- аудит списаний;
- история использования лимитов;
- основа для будущих тарифов.

## 8. Политики доступа

Обязательные правила:

- пользователь видит только свои `children`;
- пользователь видит только свои `stories`;
- пользователь не может читать/изменять чужие записи;
- доступ к файлам в Storage должен быть ограничен владельцем или выдаваться через подписанные ссылки.

## 9. Серверная архитектура

Рекомендуемое разделение:

- `app/` — страницы и UI
- `components/` — переиспользуемые компоненты
- `lib/supabase/` — клиенты Supabase
- `lib/ai/` — генерация текста
- `lib/tts/` — генерация аудио
- `lib/validators/` — схемы валидации
- `app/actions/` или `features/*/actions` — Server Actions
- `app/api/` — webhooks и route handlers

## 10. Логика генерации

Рекомендуемый пайплайн:

1. Создать запись `stories` со статусом `pending`.
2. Проверить баланс пользователя.
3. Сгенерировать текст.
4. Сохранить текст и статус `text_ready`.
5. Сгенерировать аудио.
6. Загрузить аудио в Storage.
7. Обновить запись: `audio_url`, `status = completed`.
8. Создать запись в `usage_events`.
9. Уменьшить `stories_balance`.

Если ошибка на шаге генерации текста:

- `status = failed`
- баланс не списывать

Если ошибка на шаге генерации аудио:

- текст сохранить
- `status = failed` или отдельный `audio_failed`
- баланс лучше не списывать в MVP, либо списывать только если пользователь получил текст; это надо зафиксировать отдельно.

## 11. Интеграции

### 11.1 LLM

Требования:

- таймаут запроса;
- retry на временных ошибках;
- серверная валидация ответа;
- защита от пустого или слишком короткого результата.

### 11.2 TTS

Требования:

- таймаут;
- ограничение максимальной длины текста;
- обработка неуспешной генерации без потери текста сказки.

### 11.3 YooKassa

Не включать полную подписочную механику в первый этап, пока не будет подтвержден спрос.

Если платежи все же нужны:

- отдельные таблицы `subscriptions`, `payments`, `payment_events`;
- webhook handler;
- idempotency key;
- статусная модель платежей.

## 12. UI-экраны MVP

Обязательные страницы:

- `/` — лендинг
- `/auth/login`
- `/auth/sign-up`
- `/dashboard`
- `/children`
- `/children/new`
- `/stories/new`
- `/stories/[id]`

На странице создания сказки:

- выбор ребенка;
- ввод темы;
- кнопка запуска;
- индикатор статуса.

На странице сказки:

- название;
- текст;
- аудиоплеер;
- дата создания;
- статус.

## 13. Валидация

Формы должны проверять:

- обязательные поля;
- длину строк;
- диапазон возраста;
- недопустимые пустые значения;
- защиту от слишком длинной темы.

Рекомендуется `zod`.

## 14. Требования к качеству кода

- Чистая модульная структура.
- Разделение UI и серверной логики.
- Типизация всех входов/выходов.
- Обработка всех ожидаемых ошибок.
- Без дублирования логики.
- Server-only для секретных модулей.
- Не использовать клиентские компоненты без необходимости.

## 15. Ограничения и безопасность

- Не хранить API-ключи в клиентском коде.
- Не отправлять чувствительные данные ребенка в логи.
- Не обещать медицинский, терапевтический или клинический эффект.
- В интерфейсе и оферте использовать формулировку: `сказки для эмоциональной поддержки и мягкого воспитательного сценария`, а не `лечение` или `терапия`.

## 16. Критерии готовности MVP

MVP считается готовым, если:

- пользователь может зарегистрироваться;
- может создать профиль ребенка;
- может сгенерировать сказку по теме;
- получает текст и аудио;
- история сказок сохраняется;
- данные изолированы через RLS;
- ошибки внешних API не ломают приложение;
- приложение работает в production-среде.

## 17. Приоритет разработки

Порядок реализации:

1. Auth + profiles
2. children CRUD
3. stories schema + RLS
4. генерация текста
5. генерация аудио
6. история сказок
7. лимиты
8. платежи

## 18. Что убрать из исходного ТЗ

Из первой версии я бы убрал или отложил:

- генерацию обложек;
- Midjourney/Fal.ai;
- сложные подписки;
- громкое слово `терапевтические` в продуктовой логике;
- синхронную генерацию всего в одном действии.
